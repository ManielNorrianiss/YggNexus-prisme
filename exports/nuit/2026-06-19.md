## 1. publish.py — revalidation des pages de regroupement

### Le probleme (rappel de la memoire)
`publish.py` ne revalidait que `/` et `/tools/<slug>`. Les pages `/best/<cat>` et
`/categories/<cat>` ont `revalidate = 86400` (24 h) cote Next.js. Apres une
reclassification, elles montraient une vieille photo jusqu'a 24 h (ou redeploy manuel).

### Ce que j'ai change
Dans `main()` de `publish.py` :
- `paths` demarre maintenant a `["/", "/categories"]` et on accumule un set `cat_slugs`.
- Dans la boucle d'upsert, chaque categorie d'un outil est ajoutee a `cat_slugs`.
- Apres la boucle, on lit **toutes** les categories existantes dans Supabase
  (`select slug from categories`) et on les fusionne dans `cat_slugs`, puis on ajoute
  `/best/<slug>` et `/categories/<slug>` aux `paths` a revalider.
- Un print resume le nombre de chemins (dont X categories).

### Decision (et pourquoi)
**J'ai choisi de revalider TOUTES les categories existantes, pas seulement celles des
outils publies dans le run.** Raison : couvrir le cas du **retrait** de categorie. Quand
un outil sort d'une categorie (reclassification), il ne figure plus dans la liste de
cette categorie — mais sa vieille page de regroupement doit quand meme etre rafraichie.
Se limiter aux categories "touchees" raterait ce cas. Les categories sont peu nombreuses
(7), donc le cout est negligeable. La lecture Supabase est enrobee d'un `try/except` :
si elle echoue, on retombe proprement sur les seules categories touchees.

C'est une lecture seule de plus cote Supabase (un `select slug`), aucun risque d'ecriture.

### Verification
- `python -m py_compile publish.py` -> **OK**.
- Revue du diff : 3 inserts cibles, ancres uniques verifiees par assert avant ecriture
  (regle "montage Windows tronque" respectee : edite via bash/python, pas Edit/Write).

   Supabase, puis revalide les pages touchees (`/compare/<key>`, `/tools/<slug>/alternatives`).
   Flags : `--dry-run`, `--only compare|alternatives`, `--only-key`, `--force`, `--model`.
   Structure calquee sur `b6_seo.py`.
3. **`run_nightly.py`** : nouvelle etape `pages` (non critique) inseree apres `seo`,

4. **Frontend** (`frontend/`) :

   - `lib/queries.ts` : `getPageSeo(pageType, pageKey)` — **resilient** : retourne `null`
     si la table n'existe pas encore (migration non appliquee) ou en cas d'erreur, donc
     le site ne casse pas si on deploie le frontend avant d'avoir lance la migration.
   - `app/compare/[pair]/page.tsx` et `app/tools/[slug]/alternatives/page.tsx` : rendu de
     `intro_md` sous le titre + `seo_title`/`seo_description` dans les metadonnees (fallback
     sur l'ancien texte si pas de prose).
5. **Tests** : `prisme/tests/test_b6_pages.py` (16 tests purs : compare_key, validate_intro
   anti-mince + fallbacks, content_hash compare symetrique/sensible, content_hash
   alternatives). Suite totale : **55/55 verts**.

## Decisions (et pourquoi)

- **Table generique `page_seo`** (un type + une cle) plutot que deux tables dediees :
  meme garde-fous, meme code de lecture, extensible a d'autres pages plus tard.
- **Cle compare canonique** (slugs tries alpha, `a-vs-b`) cote batch ET cote frontend :
  une seule ligne par paire quel que soit l'ordre de l'URL.
- **Lecture frontend resiliente (null si table absente)** : decouple l'ordre de
  deploiement. On peut pusher le frontend maintenant ; tant que la migration n'est pas
  passee, les pages s'affichent comme avant (sans intro), zero crash.
- **Revalidation depuis le batch** (et non depuis publish.py) : c'est `b6_pages.py` qui
  sait quelles pages de regroupement il a touchees.
- **content_hash inclut les champs des outils** (nom, desc, pricing, score) : la prose se
  regenere automatiquement si un outil change, pas seulement si la liste change.
